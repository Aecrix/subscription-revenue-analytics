# Analytics Warehouse Design

## Project

**Subscription Revenue Intelligence Platform**

**Business Domain:** FlowSync (Fictional B2B SaaS Company)

---

# Overview

This project uses a **Star Schema** to organize data for analytics and reporting.

The raw datasets generated by the ETL pipeline are suitable for data processing but not for analytical workloads. To support business intelligence, reporting, and dashboarding, the processed data is modeled into a warehouse that separates descriptive information from measurable business events.

The warehouse is designed to support common SaaS business metrics, including:

* Monthly Recurring Revenue (MRR)
* Annual Recurring Revenue (ARR)
* Customer Churn
* Customer Retention
* Revenue by Subscription Plan
* Revenue by Industry
* Revenue by Acquisition Channel
* Customer Lifetime Value (LTV)
* Refund Analysis

The design intentionally keeps the schema simple while following analytics engineering best practices.

# Warehouse Architecture

```text
                    dim_users
                        |
                        |
                        |
            +-----------+-----------+
            |                       |
            |                       |
fact_subscriptions        fact_payments
            |
            |
            |
        dim_plans
```

The warehouse consists of:

* 2 Dimension Tables
* 2 Fact Tables

---

# Dimension Tables

Dimension tables store descriptive business information used for filtering, grouping, and segmentation.

---

## dim_users

Stores customer information.

**Primary Key**

* user_id

### Columns

| Column              | Description                  |
| ------------------- | ---------------------------- |
| user_id             | Unique user identifier       |
| customer_id         | Business customer identifier |
| signup_date         | Customer signup date         |
| signup_year         | Year extracted during ETL    |
| signup_month        | Month extracted during ETL   |
| country             | Customer country             |
| industry            | Business industry            |
| company_size        | Company size                 |
| acquisition_channel | Marketing acquisition source |
| device              | Primary device used          |

### Business Use

* Customer segmentation
* Cohort analysis
* Churn analysis
* Acquisition analysis
* Dashboard filtering

---

## dim_plans

Stores subscription plan information.

Plan attributes are separated into a dimension instead of being repeated for every subscription record.

**Primary Key**

* plan_id

### Columns

| Column        | Description                         |
| ------------- | ----------------------------------- |
| plan_id       | Plan identifier                     |
| plan_name     | Starter / Pro / Enterprise          |
| billing_cycle | Monthly or Annual                   |
| monthly_price | Standard monthly subscription price |

### Business Use

* Revenue by plan
* Plan performance
* Subscription distribution

---

# Fact Tables

Fact tables store measurable business events.

---

## fact_subscriptions

Stores one record per subscription.

**Primary Key**

* subscription_id

**Foreign Keys**

* user_id → dim_users
* plan_id → dim_plans

### Columns

| Column                   | Description                          |
| ------------------------ | ------------------------------------ |
| subscription_id          | Subscription identifier              |
| user_id                  | Customer reference                   |
| plan_id                  | Subscription plan reference          |
| start_date               | Subscription start date              |
| end_date                 | Subscription end date                |
| subscription_status      | Current subscription status          |
| discount_pct             | Applied discount percentage          |
| final_monthly_price      | Effective monthly subscription price |
| subscription_length_days | Subscription duration in days        |

### Business Use

* Active subscriptions
* Churn calculation
* Retention analysis
* Plan adoption
* Subscription growth

---

## fact_payments

Stores every payment transaction.

Each subscription can generate multiple payment records over time.

**Primary Key**

* payment_id

**Foreign Key**

* subscription_id → fact_subscriptions

### Columns

| Column          | Description                                      |
| --------------- | -------------------------------------------------|
| payment_id      | Payment identifier                               |
| subscription_id | Subscription reference                           |
| payment_date    | Payment date                                     |
| amount          | Payment amount                                   |
| payment_status  | Payment status                                   |
| refund_flag     | Indicates whether the payment was refunded       |
| payment_month   | Year-Month (YYYY-MM) extracted from payment_date |

### Business Use

* Revenue analysis
* MRR and ARR calculations
* Refund analysis
* Payment success rate
* Revenue trends

* One Plan → Many Subscriptions
* One Subscription → Many Payments

# Design Decisions

The warehouse follows a star schema because it provides:

* Simple analytical queries
* Easier aggregation
* Better Tableau integration
* Clear separation between descriptive attributes and measurable events
* Reduced query complexity

This structure keeps business logic focused on the warehouse instead of the operational datasets.

---

# Project Scope

The warehouse currently includes the datasets required for core SaaS analytics:

* Users
* Subscription Plans
* Subscriptions
* Payments

The `engagement_logs` dataset remains in the processed data layer. It is intentionally excluded from the warehouse because it is not required for the core business metrics implemented in this project. This keeps the warehouse focused and avoids unnecessary complexity.

---

# Planned Business Metrics

The warehouse is designed to support the following SQL analyses:

* Monthly Recurring Revenue (MRR)
* Annual Recurring Revenue (ARR)
* Customer Churn Rate
* Customer Retention
* Customer Lifetime Value (LTV)
* Revenue by Plan
* Revenue by Industry
* Revenue by Acquisition Channel
* Monthly Revenue Trends
* Refund Rate
* Cohort Analysis
* Executive KPI Dashboard

Business metrics will be calculated in SQL rather than during ETL so that the ETL layer remains responsible only for preparing clean, standardized data while analytical logic resides in the warehouse.
